## Overview

Audit logs record every administrative action and denied access in Bifrost Enterprise. Each entry captures who did what, when, to which resource, and whether it was allowed, denied, or errored. Entries are stored in the enterprise database (`ent_audit_entries`) and are read-only from the UI.

This page describes the **currently shipping** audit-logs surface.

### What's in scope today

| Capability | Status |
|---|---|
| Append-only storage (`ent_audit_entries`) | ✓ |
| List with filters (actor / action / resource / outcome / date range) | ✓ |
| CSV and JSON export of filtered results | ✓ |
| RBAC gate — requires `AuditLogs.View`; `AuditLogs.Download` for export | ✓ |

### What's **not** in this release

The following are commonly requested extensions; each requires its own feature spec before implementation (per SR-01 "reuse-over-new"):

- SIEM integrations (Splunk / Datadog / Elastic push)
- Webhook destinations for audit events
- Compliance-framework report generation (SOC 2, ISO 27001, etc.)
- Cryptographic / HMAC tamper verification
- Severity classification
- Retention-based automatic cleanup (rows accumulate indefinitely today)

If your deployment needs any of the above, open a request — it's a deliberate scoping line, not a plan.

## What gets logged

Every administrative action emitted through `audit.Emit()` writes an entry with:

| Field | Contents |
|---|---|
| `actor_type` | `user` / `admin_api_key` / `service_account` / `system` |
| `actor_id`, `actor_display`, `actor_ip` | Caller identity |
| `action` | Canonical verb, e.g. `role.create`, `workspace.delete`, `guardrail.deny` |
| `resource_type`, `resource_id` | Affected resource |
| `outcome` | `allowed` / `denied` / `error` |
| `reason` | Free-text detail for non-allowed outcomes |
| `before_json`, `after_json` | Optional state snapshots |
| `request_id` | For correlation with request logs |
| `organization_id`, `workspace_id` | Tenant scoping |
| `created_at` | UTC timestamp (indexed) |

Events are emitted from enterprise-aware handlers as admin actions happen. Upstream inference paths are not audited here — use request logs for those.

## Viewing audit logs

Navigate to **Audit Logs** in the Bifrost workspace sidebar.

Available filters:

- **Actor ID** — exact match on the user or API-key UUID
- **Action** — e.g. `role.create`, `workspace.update`
- **Resource type** — e.g. `role`, `user`, `workspace`
- **Outcome** — `allowed` / `denied` / `error`
- **From / To** — ISO 8601 date-time range
- **Limit / Offset** — paginate up to 500 per page (default 50)

Apply the filters, then use the **CSV** or **JSON** export button to download the filtered set for offline analysis or out-of-band SIEM ingestion.

## API

All endpoints require a session with `AuditLogs.View`; the export endpoints additionally require `AuditLogs.Download`.

### List

```bash
curl -s "http://localhost:8088/api/audit-logs?outcome=denied&limit=50&offset=0"
```

Response:

```json
{
  "entries": [
    {
      "id": "01945d8c-...",
      "organization_id": "...",
      "workspace_id": "",
      "actor_type": "user",
      "actor_id": "...",
      "actor_display": "alice@example.com",
      "actor_ip": "203.0.113.42",
      "action": "role.delete",
      "resource_type": "role",
      "resource_id": "01945d7a-...",
      "outcome": "denied",
      "reason": "cannot delete built-in role",
      "before_json": "{...}",
      "after_json": "",
      "request_id": "req_abc123",
      "created_at": "2026-04-20T08:10:12.345Z"
    }
  ],
  "total": 1,
  "limit": 50,
  "offset": 0
}
```

### Export

```bash
# CSV (filtered by same query params as the list endpoint)
curl -s "http://localhost:8088/api/audit-logs/export?format=csv&outcome=denied" \
  -o audit-denied.csv

# JSON array
curl -s "http://localhost:8088/api/audit-logs/export?format=json" \
  -o audit.json
```

Both endpoints honor the filter params from the list endpoint. No pagination on export — the entire filtered set is returned.

## Querying from the database

`ent_audit_entries` lives in the Bifrost configuration database (SQLite by default, Postgres when configured). Standard SQL works for ad-hoc investigations:

```sql
SELECT action, resource_type, outcome, COUNT(*)
FROM ent_audit_entries
WHERE created_at >= datetime('now', '-24 hours')
GROUP BY action, resource_type, outcome
ORDER BY 4 DESC;
```

Indexes on `(organization_id, created_at)`, `action`, `resource_type + resource_id`, `actor_type`, `actor_id`, `outcome`, `request_id` keep common admin queries fast.
