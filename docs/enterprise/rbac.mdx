## Overview

Role-Based Access Control (RBAC) in Bifrost Enterprise controls what each user can do. Permissions are expressed as `Resource.Operation` scopes (e.g. `VirtualKeys.Create`, `AuditLogs.Download`); a **role** is a named collection of those scopes, and a **role assignment** grants a role to a user.

**Key properties:**

- **24 resources × 6 operations** — every enterprise-visible concept is a resource; every write-shaped action is an operation.
- **4 built-in roles** seeded at first boot: Owner, Admin, Manager, Member.
- **Custom roles** are fully editable; built-in roles are read-only.
- **Per-user assignments** — each user receives one or more roles; assignments can be org-wide or workspace-scoped.
- Every role change and assignment change emits an audit entry (see [Audit Logs](./audit-logs)).

---

## Resources & operations

RBAC is structured as a matrix. The 24 resources currently defined:

`GuardrailsConfig`, `GuardrailsProviders`, `GuardrailRules`, `UserProvisioning`, `Cluster`, `Settings`, `Users`, `Logs`, `Observability`, `VirtualKeys`, `ModelProvider`, `Plugins`, `MCPGateway`, `AdaptiveRouter`, `AuditLogs`, `Customers`, `Teams`, `RBAC`, `Governance`, `RoutingRules`, `PIIRedactor`, `PromptRepository`, `PromptDeploymentStrategy`, `AccessProfiles`.

The 6 operations:

| Operation | Meaning |
|---|---|
| `Read` | Read raw data (counts, list endpoints for downstream use) |
| `View` | View in UI (list/detail rendering) |
| `Create` | Create a new instance |
| `Update` | Modify an existing instance |
| `Delete` | Remove an instance |
| `Download` | Export data off-system (CSV/JSON) |

Not every resource meaningfully supports every operation (e.g. `AuditLogs` is append-only — `Create`/`Update`/`Delete` don't apply even if the scope is granted). The matrix is intentionally wide so a single role can cover a whole area without operation-specific surprises.

## Built-in roles

| Role | Intended holder | Notes |
|---|---|---|
| **Owner** | Organization owner | Full access — wildcard scope `*`. Cannot be modified or deleted. |
| **Admin** | Platform admin | Broad admin access on Customers, Teams, VirtualKeys, Guardrails, AuditLogs, RBAC, etc. — one step below Owner. |
| **Manager** | Workspace admin | Workspace-scoped admin (Teams, Users, RoutingRules, PromptRepository) plus view access to governance. |
| **Member** | Default end-user | Read/view across most resources; limited write access on Prompts. |

The exact scope set for each built-in role lives in `framework/configstore/migrations_enterprise.go` (`builtInRoles`), seeded by the `E004_orgs_workspaces_users_roles` migration at first boot. Built-in roles can be **viewed** in the UI but not edited or deleted — to customize, clone their intent into a new custom role.

## Managing roles & users from the UI

All management surfaces are under `/workspace/governance/`:

- **`/workspace/governance/rbac`** — roles list; create / edit / delete custom roles. The editor presents the full 24×6 matrix with a row-click shortcut to toggle all operations for a resource at once.
- **`/workspace/governance/users`** — users list; invite a new user by email, manage role assignments per-user (assign, unassign, see current roles).

The UI gates itself: a user without `RBAC.View` cannot open the roles page, without `Users.Create` cannot invite users, etc. The gates reflect the user's resolved scopes fetched from `/api/rbac/me` at session start.

## API

All endpoints require a session with the appropriate scope on the `RBAC` and `Users` resources.

### Meta & current user

```bash
# 24 resources + 6 operations + the built-in role names
curl -s http://localhost:8088/api/rbac/meta

# Current user's resolved permissions (used by the UI's RbacProvider)
curl -s http://localhost:8088/api/rbac/me
```

### Roles

```bash
# List
curl -s http://localhost:8088/api/rbac/roles

# Create custom
curl -s -X POST http://localhost:8088/api/rbac/roles \
  -H 'Content-Type: application/json' \
  -d '{
    "name": "Auditor",
    "scopes": {
      "AuditLogs": ["Read", "View", "Download"],
      "Logs": ["Read", "View"]
    }
  }'

# Update (only custom roles)
curl -s -X PATCH http://localhost:8088/api/rbac/roles/{id} \
  -H 'Content-Type: application/json' \
  -d '{"scopes": {"AuditLogs": ["Read", "View", "Download"], "Logs": ["Read", "View", "Download"]}}'

# Delete (only custom roles)
curl -s -X DELETE http://localhost:8088/api/rbac/roles/{id}
```

Attempting to modify or delete a built-in role returns `403 cannot modify built-in role`.

### Users

```bash
# List (includes active role assignments per user)
curl -s http://localhost:8088/api/rbac/users

# Invite (creates a pending user; activated on first successful SSO login)
curl -s -X POST http://localhost:8088/api/rbac/users \
  -H 'Content-Type: application/json' \
  -d '{"email": "alice@example.com", "display_name": "Alice"}'

# Update
curl -s -X PATCH http://localhost:8088/api/rbac/users/{id} \
  -H 'Content-Type: application/json' \
  -d '{"status": "suspended"}'

# Delete (also removes all their role assignments)
curl -s -X DELETE http://localhost:8088/api/rbac/users/{id}
```

### Assignments

```bash
# List assignments for a user
curl -s http://localhost:8088/api/rbac/users/{userId}/assignments

# Assign a role (optionally workspace-scoped)
curl -s -X POST http://localhost:8088/api/rbac/assignments \
  -H 'Content-Type: application/json' \
  -d '{"user_id": "...", "role_id": "...", "workspace_id": ""}'

# Unassign
curl -s -X DELETE http://localhost:8088/api/rbac/assignments/{assignmentId}
```

Workspace-scoped assignments apply only when the user is acting in that workspace; an empty `workspace_id` means org-wide.

## Examples

### A read-only auditor

```json
{
  "name": "Auditor",
  "scopes": {
    "AuditLogs": ["Read", "View", "Download"],
    "Logs":      ["Read", "View"],
    "Users":     ["Read", "View"],
    "RBAC":      ["Read", "View"]
  }
}
```

### A workspace operator (governance-focused)

```json
{
  "name": "Ops Lead",
  "scopes": {
    "VirtualKeys":  ["Read", "View", "Create", "Update", "Delete"],
    "Teams":        ["Read", "View", "Create", "Update"],
    "Governance":   ["Read", "View", "Create", "Update"],
    "RoutingRules": ["Read", "View", "Create", "Update", "Delete"],
    "Logs":         ["Read", "View", "Download"]
  }
}
```

## Notes & limits

- **Enforcement surface:** UI elements are gated today; server-side enforcement of individual scope checks on each admin handler is rolled out alongside new handlers — not every legacy upstream endpoint performs scope checks yet. Treat the enforcement line as **RBAC-aware for enterprise surfaces; permissive for upstream endpoints already exposed to the single admin basic-auth credential**.
- **SSO / SCIM auto-provisioning of role assignments is not in scope for this release.** Today, invite a user by email in the UI, then assign roles manually. SSO-based auto-mapping is a separate feature that needs its own spec before implementation.
- **Built-in role scope sets are fixed at the `builtInRoles` table in `framework/configstore/migrations_enterprise.go`**. Customizing a built-in is done by cloning its scopes into a new custom role and assigning that role instead.
